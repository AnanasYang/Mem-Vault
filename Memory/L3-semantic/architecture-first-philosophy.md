# 架构优先的设计哲学

在解决问题时，倾向于首先建立清晰的概念框架和结构体系，将具体实现置于系统架构的理解之后。

---

## 框架定义

### 核心命题
"好的设计始于清晰的架构，实现只是架构的自然延伸。"

### 底层假设
1. 系统的长期可维护性取决于架构的清晰度
2. 前期架构投入能够减少后期的重构成本
3. 结构化的思考能够降低认知复杂度
4. 明确的边界是协作效率的基础

### 适用范围
- 软件系统设计
- 工具链和流程设计
- 复杂问题的分析和拆解
- 知识管理和信息组织

---

## 构成要素

### 关键概念
- **架构视图**: 从宏观层面理解系统的组成部分和交互关系
- **分层抽象**: 将复杂系统分解为层次化的抽象层
- **接口契约**: 组件间通过明确的接口进行交互
- **一致性原则**: 在结构和命名上保持统一规范

### 原则/规则
- **先结构后细节**: 在深入具体实现前，先建立整体结构
  - 优先级: P0
  - 适用条件: 任何需要长期维护的系统
- **边界先于实现**: 明确模块边界后再填充实现
  - 优先级: P1
  - 适用条件: 多模块协作的场景
- **可扩展性预设**: 架构设计时考虑未来可能的变化
  - 优先级: P2
  - 适用条件: 预计会演进的系统

### 决策流程
```
1. 理解问题空间和约束条件
2. 识别核心概念和实体
3. 建立分层/分模块的架构草案
4. 定义模块间的接口和依赖关系
5. 评估架构的清晰度和可扩展性
6. 迭代优化架构
7. 进入具体实现阶段
```

## 应用实例

### 场景1: AI记忆系统设计
**背景**: 设计5层记忆架构
**决策过程**: 
1. 先定义各层职责边界
2. 设计层间的数据流向
3. 确定每层的review周期
4. 最后才创建具体文件
**结果**: 架构清晰，便于后续迭代和维护

### 场景2: 多智能体系统
**背景**: 设计多个Agent协作的系统
**决策过程**:
1. 绘制Agent交互架构图
2. 定义消息传递协议
3. 明确各Agent的职责边界
4. 最后实现具体Agent逻辑
**结果**: 系统扩展性好，新增Agent无需改动核心

### 反事实分析
- **如果不遵循此框架**: 可能直接开始编码，后期发现结构问题需要大量重构
- **对比**: 前期架构投入约占总时间20%，但减少了后期60%的重构工作

## 边界与限制

### 适用边界
- 适合需要长期维护的系统
- 适合多人协作的项目
- 适合复杂度较高的设计任务

### 不适用场景
- 探索性/实验性工作（初期）
- 一次性脚本或临时任务
- 时间压力极大且明确的一次性交付

### 潜在盲点
- 可能过度设计简单问题
- 在快速变化的需求面前，架构可能过早固化
- 对"前期投入"的坚持可能导致交付延迟

## 框架变体

### 强形式
在所有项目中强制要求完整的架构设计文档和评审流程

### 弱形式
在脑中快速建立概念模型，不强制输出文档，但保持结构化思考

### 上下文变体
- **个人项目**: 弱形式，架构在脑中
- **团队协作**: 强形式，需要文档化
- **紧急修复**: 可暂时跳过，但后续补充架构文档

## AI应用指南

### 何时引用此框架
- 用户要求设计系统或架构时
- 用户询问"应该怎么做"而不是"具体怎么做"时
- 涉及长期维护和扩展性考量时

### 如何应用
1. 先询问或分析整体架构需求
2. 提供结构化的架构方案
3. 强调各部分的边界和关系
4. 在架构确认后再进入实现细节

### 与其他框架的协同
- 与[效率优先原则]协同: 架构优先是为了长期效率
- 与[模块化思维]协同: 架构设计自然产生模块化结构

### 需要谨慎的情况
- 用户明确表示"先跑起来再说"
- 时间压力极大的场景
- 用户已经提供了明确的实现方案，不需要架构讨论
